# Design: 整局游戏界面指导（屏谱优先）

日期: 2026-07-25  
状态: 已确认，待实施  
形态: 在现有 Agent Plugin + `library/packs` 上，把产品叙事与工作流从「风格/组件库」升级为「整局界面指导」

**关联：** 双路径（有 pack / 自建）、统一共享栈、无 `examples/generic` 场景。

---

## 1. 目标

本仓库对外定位改为：

> **竖屏触控游戏的整局界面指导**（信息架构 + 屏谱 + 玩法楼型 + 视觉风格），附 10 套类型正确的可运行参考 pack。

组件、Token、共享 CSS 是**落地手段**，不是第一叙事。

**非目标（本期）**

- 不要求每个 pack 立刻做出全部次级屏的完整可玩 HTML  
- 不重做 10 套主场景玩法  
- 不恢复「通用 HUD 范例」目录  

---

## 2. 已确认决策

| 决策 | 选择 |
|------|------|
| 范围 | **C**：屏谱/导航 + 整界面方法论 + Agent 先循环/屏谱后风格 |
| 屏谱约束 | **按类型推荐、可裁剪**（非全游戏强制同一最小集）；矩阵写清该类型默认要哪些 |
| 无对应 pack | 路径 B：按该游戏类型自建楼型与视觉；无仓库内通用场景 |
| 主场景 | 仍可只演示**核心玩法循环**；全屏谱以 Spec / 规则文为准 |

---

## 3. Agent 工作流（权威顺序）

1. **定游戏** — 类型、核心循环、情绪基调  
2. **定屏谱** — 需要哪些屏、常驻 vs 模态、进出关系  
3. **定主 HUD 楼型** — 矩阵行或自建；保证盲测循环  
4. **定风格路径** — 有对应 → pack（A）；无对应 → 自建（B）  
5. **落地** — `library-map` 加载序；中文；认形；反网页壳  
6. **验收** — 屏谱相对完整 + HUD 循环 + `audit-game-hud`  

禁止：未定循环/屏谱就先贴 pack 换皮。

---

## 4. 屏类型词汇（跨游戏词典）

下列为**推荐清单**；是否启用由游戏类型决定（见矩阵扩展列）。

| 屏/层 | 职责 | 常见形态 |
|-------|------|----------|
| 标题 / 主菜单 | 开始、继续、设置入口 | 全屏或半屏栈 |
| 主玩法 HUD | 核心循环操作与状态 | 常驻边锚 / 底栏 / 指令窗等 |
| 暂停 | 离开战斗空间的中断 | 模态 |
| 设置 | 音量、语言、操作说明 | 模态或子页 |
| 背包 / 角色 / 构筑 | 管理资源或成长（按类型） | 模态或半屏 |
| 对话 / 提示 / 日志 | 叙事与情境文案 | 条、窗、低语 |
| 结算 / 失败再开 | 一局结束与重开 | 全屏或叠层 |
| 商店 / 地图 / 导航 | 类型可选 | 按矩阵 |

**原则：** 世界在先；常驻认形；中文给情境与菜单；不要用网页壳串起多屏。

## 5. 仓库改动清单

### 5.1 新规则文（权威）

创建 `docs/rules/game-interface-system.md`：

- 整局界面一句话定位  
- 屏类型词典与常驻/模态规则  
- 与 `mobile-portrait-first`、`game-feel-vs-web` 的关系  
- Agent 工作流六步（指向 Skill）  
- 反例：无屏谱先套皮；全游戏同一动作条；用落地页信息架构串屏  

### 5.2 叙事重写

- `README.md` / `CLAUDE.md` / `INSTALL.md` / `.claude-plugin/plugin.json`  
  - 第一句改为整局界面指导  
  - 组件降为实现附录  

### 5.3 Skill / Commands

- 扩展 `skills/game-ui-style/SKILL.md`（或拆 `game-interface` 主 Skill + 风格为子步骤）：**强制先屏谱后风格**  
- 新增 `commands/map-screens.md`：输出该游戏的屏谱表  
- `pick-style` / `apply-style`：前置「屏谱已定或本步先定」  
- `audit-game-hud`：增加「屏谱」检查项（相对类型推荐是否合理、主循环屏是否可盲测）  

### 5.4 矩阵与 references

- `references/style-matrix.md` + `docs/rules/mobile-portrait-first.md` §4：增加 **推荐屏谱** 列（可裁剪说明）  
- `references/hard-rules.md`：链到新规则文；强调屏谱优先  

### 5.5 Pack SPEC 模板扩展

在 13 章模板中、于「适用游戏类型」之后或专设：

**「界面系统（屏谱）」** — 该风格默认屏列表、主场景覆盖哪一环、其余屏仅规范不强制 Demo。

实施可：先改 `styles` 已迁入的 `SPEC.md` 模板说明 + 抽 2 套示范（如 04、02），再批次补齐 10 套。

### 5.6 不改动的部分

- pack 目录同构（SPEC / tokens / scene.css / scene.js / index.html）  
- 共享栈加载序  
- 禁止恢复 `examples/generic`

## 6. 分期实施

| 阶段 | 内容 | 完成标准 |
|------|------|----------|
| **P0** | `game-interface-system.md` + README/CLAUDE/INSTALL/plugin 叙事 + Skill 六步工作流 | 打开 README 能读出「整局指导」；Skill 禁止先套皮 |
| **P1** | `/map-screens` + 矩阵「推荐屏谱」列 + hard-rules/audit 挂钩 | pick/apply 要求屏谱；audit 含屏谱项 |
| **P2** | 10 份 SPEC 补「界面系统」；至少 2 套写细、其余按模板填默认屏谱 | 每 pack SPEC 可回答「默认有哪些屏」 |

---

## 7. 成功标准

1. 对外叙事不再以「组件库」为第一句。  
2. Agent 默认顺序为：游戏 → 屏谱 → HUD 楼型 → 风格路径 → 落地。  
3. 类型矩阵可查「该类型推荐屏谱」。  
4. 无对应 pack 时仍产出类型正确的整局界面方案（规范层），而非假通用皮肤。  
5. 现有 10 主场景玩法不被本设计破坏。  

---

## 8. 风险

| 风险 | 缓解 |
|------|------|
| Spec 膨胀 | 「界面系统」一节用表格式，禁止长文重复矩阵 |
| 与「主场景只做核心循环」冲突 | 明文：Demo 可子集；Spec 写全推荐屏 |
| Agent 仍先贴皮 | Skill 硬门：未输出屏谱表不得进入 apply pack |
