# 游戏 UI 设计大全 — 项目规范

## 目标

蒸馏多套**可程序化生成**的游戏 UI 风格文档。每套风格定义参数化规则,任何工程师拿规则即可生成符合风格的 UI(按钮/面板/进度条/弹窗/血条/小地图/背包格...)。**只定义风格,不涉及具体美术素材**(icon、character art、logo)。

## 风格文档模板

每个 `styles/*.md` 必须包含以下章节(顺序固定):

1. **风格定位** — 一句话定位 + 代表作参考
2. **视觉关键词** — 10-15 个核心形容词
3. **色彩系统** — 主色/辅色/强调色/语义色(HEX 或 HSL),含配色原则
4. **几何与构图** — 圆角、间距、对齐、网格、比例
5. **边框与容器** — 面板、按钮、卡片的具体构造规则(参数化)
6. **文字与排版** — 字体倾向、字重、间距、对齐
7. **光影与质感** — 阴影、辉光、纹理、噪点、渐变
8. **动效原则** — 入场/退出/状态切换的曲线与时长
9. **状态语义** — normal/hover/active/disabled 的视觉差
10. **可程序化生成参数表** — 一张表,列名/类型/范围/默认值/说明
11. **不做什么** — 反例,避免风格跑偏
12. **可生成部件清单** — 该风格下哪些 UI 部件用此风格渲染

## 约束

- 风格文档必须**纯参数化**:不引用具体素材文件名、不嵌入位图、不依赖字体文件路径
- 任何颜色用 HSL 或 HEX 数值,不写"红色"这种模糊描述
- 任何动效用曲线名(ease-in-out) + 秒数,不写"流畅"
- 每套风格至少给出 5 个**反例**(不要做什么)
- 反例的目的是划定边界,避免风格相互污染

## 验证

- 跑过不验证 = 没做
- 每写完 3 套风格,回头检查模板完整性
- 写完所有风格,做一次交叉对比,确保风格之间**互斥不重叠**
